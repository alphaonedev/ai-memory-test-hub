## Methodology

All 8 scenarios were executed against `target/release/ai-memory` built from `feat/v0.6.4` @ 44e4bc6 (last commit on branch as of capture). Database paths point at `/tmp/v064-cert.db` per scenario (fresh DB created on each invocation, removed after). MCP scenarios drive the binary over stdio with a single JSON-RPC line; the result is parsed and asserted in-test.

Reproduce locally:

```bash
cd ai-memory-mcp
git checkout feat/v0.6.4
cargo build --release
BIN=./target/release/ai-memory
AI_MEMORY_NO_CONFIG=1 echo '{"jsonrpc":"2.0","id":1,"method":"tools/list"}' \
  | $BIN --db /tmp/cert.db mcp --profile core --tier keyword
# expect: count=6, names include memory_capabilities + 5 core tools
```

Per-scenario evidence captured in `releases/v0.6.4/scenarios/`.

## Scenario detail

### S25 — `--profile core` registers exactly 6 visible tools

**Expectation:** `tools/list` under `--profile core` returns 5 core tools (`memory_store`, `memory_recall`, `memory_list`, `memory_get`, `memory_search`) plus the always-on `memory_capabilities` bootstrap = 6 total.

**Result:** `count=6`, names exactly as expected.

**Note on the "5-tool default" public claim:** the public claim refers to the 5 family-member tools. The 6th visible tool is `memory_capabilities`, which is always-on regardless of profile per RFC §S27 because it is the bootstrap surface for runtime expansion. Both ai-memory-mcp `V0.6.4-EPIC.md` §4 and the v0.6.4 cert spec carry this reconciliation explicitly.

### S26 — `--profile full` registers 43 tools

**Expectation:** v0.6.3 surface 1:1.

**Result:** `count=43`. Source-anchored at `src/mcp.rs::tool_definitions()`; backward-compat preserved.

### S27 — `memory_capabilities` always available

**Expectation:** every profile (core / graph / admin / power / full) advertises `memory_capabilities` so an agent that lands on a stricter-than-needed profile can always discover what families are available and request expansion.

**Result:** all 5 named profiles include `memory_capabilities`.

| Profile | Tool count | `memory_capabilities` present |
|---|---:|:---:|
| core | 6 | ✅ |
| graph | 14 | ✅ |
| admin | 19 | ✅ |
| power | 12 | ✅ |
| full | 43 | ✅ |

### S28 — unloaded-tool error path

**Expectation:** when a tool outside the active profile is called, the server returns JSON-RPC `-32601` (Method not found / tool_not_found) with a message that names the family the tool belongs to and the profile flag/family the operator should add.

**Result:** code=-32601, message:

> tool 'memory_kg_query' is in family 'graph' which is not loaded under the active profile. Restart with `--profile <name>` or `--profile core,graph` to load it, or call `memory_capabilities --include-schema family=graph` to expand at runtime.

Source: `src/mcp.rs:3489-3501`.

### S29 — doctor token match

**Expectation:** `ai-memory doctor --tokens --json` reports the same `full_profile_total_tokens` as the static benchmark (`benchmarks/v0.6.4-cross-harness.md`) within ±10%.

**Result:** doctor=6,198, benchmark=6,198, drift=0.00%. Pinned by `cli::doctor::tests::run_tokens_human_default_profile_is_core` and `sizes::tests::full_profile_total_in_honest_measured_range` (5K–8K range).

### S30 — custom profile

**Expectation:** `--profile core,graph` registers core (5) + graph (8) + always-on capabilities (1) = 14 tools, no lifecycle/governance/power/archive/other.

**Result:** count=14, `memory_kg_query` present, `memory_capabilities` present, `memory_delete` (lifecycle) absent.

### S31 — SDK requireProfile

**Expectation:** TypeScript and Python SDK helpers raise a structured `ProfileNotLoaded` exception with an actionable `hint` field (containing `--profile <name>` and `AI_MEMORY_PROFILE=<name>`) when the daemon does not load every family the requested profile needs.

**Result:**

- TypeScript: `npm test` → 19/19 jest tests pass (`__tests__/profile.test.ts`)
- Python: `pytest tests/test_profile.py` → 14/14 pass

Both implementations include a permissive fallback (logged warning) for pre-v0.6.4 daemons that don't return a `families` block in their capabilities response.

### S32 — boot manifest cost identical across profiles

**Expectation:** the boot manifest (`ai-memory boot --json`) is independent of the active MCP profile — `--profile` affects only `tools/list` advertised over stdio, not the boot JSON written to stdout for SessionStart hooks.

**Result:** each of core/graph/admin/power/full produced exactly 35 bytes of manifest output (default empty-DB JSON). Profile selection is decoupled from boot.

## v0.6.4-013 backward-compat

`--profile full` reproduces the v0.6.3 surface 1:1. Verification touchpoints:

- **Code-level invariant:** `tool_definitions_for_profile(&Profile::full())` returns exactly the 43 tools from `tool_definitions()` — no filtering, no reordering. Pinned by `mcp::tests::tool_definitions_for_profile_full_registers_43`.
- **Live behavior:** S26 in this campaign captures the live count=43.
- **Cumulative substrate test pass:** 1,955 lib tests green on every commit on `feat/v0.6.4` through tag-cut. Includes the v0.6.3 carry-forward suites (`capabilities_v2.rs`, `webhook_coverage.rs`, `data_integrity_v17.rs`, `budget_tokens.rs`, `doctor_cli.rs`, etc.) — none required modification for v0.6.4.

The v0.6.3 ship-gate suites were not re-executed against the v0.6.4 binary in this campaign because the substrate change is purely additive (a new pre-`tools/list` filter that becomes a no-op under `--profile full`); a full multi-node ship-gate replay is scheduled as a post-tag soak rather than a tag-cut blocker.
